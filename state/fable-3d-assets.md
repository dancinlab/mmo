# fable 3D game-asset implementation — mmo block-diorama look · 2026-07-02 · sidecar fable (claude-fable-5, headless) · grounded on 5 opened images

# mmo 블록 디오라마 → 실제 3D 게임 에셋 구현 컨설트

**이미지 관찰 근거 (5장 정독):** 지형·성벽·절벽은 큰 슬래브(대략 4×2×1 ~ 8×4×2 비율의 직육면체)로 매스를 잡고, 지붕·다리·가구는 중간 블록, 나뭇잎·시장 물건·랜턴·캐릭터 디테일은 1유닛급 소형 블록 클러스터로 채워져 있음. 모든 블록은 스터드/홈 없는 매끈한 면 + 미세한 모서리 베벨 + 무광 플라스틱 재질이며, 블록 경계마다 밝은 엣지 하이라이트가 있음. 캐릭터는 몸통·팔다리·머리카락까지 여러 작은 블록으로 세분화(pet-ranch의 관리인, fireworks-festival의 군중)되어 있고 큰 정육면체 머리는 없음. 받침대는 유광 검정으로 씬을 반사. 조명은 단일 웜 키라이트(골든아워 또는 달빛) + 창문/랜턴의 강한 emissive + 림라이트 + 블룸 + 틸트시프트 피사계심도. dungeon-boss의 오우거와 pet-ranch의 드래곤들은 블록 문법을 따르되 유기적 실루엣을 가진 "세미 스컬프트" 예외임 — 이 점이 파이프라인 설계에 중요.

---

## 1. 저작 파이프라인 — 권장: **복셀 저작 + 자동 "블록화(box-decomposition)" 변환 하이브리드**

**핵심 통찰:** 우리 룩의 "비균일 블록"은 손으로 제각각 크기의 박스를 배치한 결과가 아니라, **균일 복셀 그리드로 저작한 뒤 탐욕적 박스 분해(greedy box decomposition)로 병합하면 자동으로 나오는 결과물**이다. 8×4×2 복셀 덩어리를 하나의 베벨 박스로 병합하면 레퍼런스의 큰 슬래브가 되고, 병합 불가능한 디테일 영역은 자연히 작은 블록으로 남는다. 따라서 저작은 복셀로 하되 **출력이 복셀이 아니라 "베벨 박스의 집합"**이 되도록 변환 단계를 만든다.

**권장 파이프라인 (에셋 클래스별):**

- **지형·건물·구조물:** MagicaVoxel로 복셀 저작(씬 그리드 기준 건물 한 채 32³~64³) → 커스텀 변환기(`vox2blocks`)가 .vox를 읽어 ① 색상별 greedy box decomposition ② 각 박스에 모서리 베벨(챔퍼) 지오메트리 부여 ③ 팔레트 → 버텍스 컬러 매핑 ④ GLB로 출력. 병합 최대 크기에 상한(예: 8유닛)을 두고 의도적으로 분할 시드를 섞어 "블록 쌓은 느낌"의 줄눈을 유지한다 — 무제한 병합하면 그냥 로우폴리 박스 건물이 되어 룩이 죽는다.
- **소품(프롭):** 동일 파이프라인, 더 작은 그리드(8³~16³). 랜턴·술통·시장 좌판 등은 변환 후 Blender에서 emissive 머티리얼만 지정.
- **캐릭터·크리처:** MagicaVoxel로 파트별(머리/몸통/팔/다리/머리카락) 저작 → 파트 단위로 블록화 → Blender에서 조립·리깅. 단, dungeon-boss 오우거나 드래곤 같은 유기적 히어로 크리처는 **Blender에서 로우폴리 스컬프트 후 "리메쉬 투 블록"(복셀 리메셔 → 블록화)** 경로로 실루엣을 살린다.
- **AI image-to-3D(TRELLIS/Tripo/Meshy/Rodin)의 역할:** 최종 에셋으로는 **부적합** — 현 세대 모델은 블록 문법(직교 평면, 일정한 베벨, 깨끗한 줄눈)을 유지하지 못하고 뭉개진 유사-복셀 표면을 뱉는다. 단 **블록아웃/비율 레퍼런스**로는 유용: 2D 레퍼런스를 TRELLIS로 돌려 나온 러프 메시를 MagicaVoxel에 복셀화 임포트하여 매스·비율의 밑그림으로 쓰고 위에 다시 저작한다.

대안 한 줄씩: ① 순수 MagicaVoxel 유닛 복셀 출력 — 블록 크기가 균일해져 v1(rejected)의 장난감 룩으로 퇴행하므로 탈락. ② Blender에서 손으로 박스 배치 — 룩 통제는 최고지만 32씬 규모에서 공수 폭발. ③ 순수 프로시저럴 블록 어셈블리 — 건물 문법 코딩 비용이 저작 절감보다 큼, 나무/울타리 등 반복물에만 부분 적용.

## 2. 에셋 분해 — 권장: **"모듈러 킷 80% + 씬당 히어로 원오프 20%"**

32씬을 훑으면 바이옴이 사실상 5개로 수렴한다: **중세 판타지**(town-square, guild-hall, snowy-fortress, siege-battle 등), **동양/축제**(fireworks-festival, hotspring-ryokan), **자연/전원**(harvest-farm, pet-ranch, world-tree-village, pixel-forest 등), **던전/동굴**(dungeon-boss, lava-mines, mushroom-cavern, haunted-crypt), **SF/사이버**(spaceport-hangar, cyber-market). 권장 킷 분류와 개수:

| 킷 | 내용 | 개수(개) |
|---|---|---|
| 베이스/지형 | 받침대(유광), 지면 슬래브, 절벽 모듈, 물 타일, 계단·경사로, 길 포장 | ~40 |
| 건축 킷 ×5 바이옴 | 벽·지붕·문·창·기둥·탑·담장·다리 각 ~30 | ~150 |
| 자연 | 나무 3크기×4종, 덤불, 꽃, 버섯, 바위, 벚꽃/단풍 변형 | ~35 |
| 프롭 세트 | 시장/축제(좌판·천막·깃발), 공방(모루·선반), 마법(포션·룬), 항구(배럴·로프), 농장(울타리·여물통) | ~110 |
| 라이팅 프롭 | 랜턴 4종, 횃불, 화로, 네온 사인, 가로등 — 전부 emissive 슬롯 보유 | ~15 |
| 캐릭터 | 모듈러 휴머노이드 1베이스(머리·헤어·상의·하의·손도구 파츠 각 8~10종) + 크리처 12종(슬라임, 베이비드래곤, 대형 드래곤, 오우거, 가축, 물고기, 유령, 로봇) | 베이스 1 + 파츠 ~45 + 크리처 12 |
| FX/셰이더 | 폭포, 불꽃놀이 파티클, 포탈, 안개 카드, 물결, 용암, 스팀 | ~10 |

**합계 약 400피스**로 32씬의 80%를 커버. 씬당 히어로 원오프(세계수 본체, 갤리온선, 오우거, 스타쉽 등) 3~5개는 개별 저작. 씬 자체는 킷 인스턴스 배치를 기록한 **씬 레이아웃 JSON**으로 정의하여, 디오라마 = "데이터", 킷 = "코드"가 되게 한다.

대안 한 줄: 씬별 통짜 원오프 저작 — 첫 3씬은 빠르지만 32씬 총공수와 용량이 3~4배, 마켓플레이스 에셋 재판매도 불가하므로 탈락.

## 3. 테크 스펙 — 권장: **GLB 단일 포맷 · three.js 마켓 프리뷰 · Godot 4 게임 런타임**

- **엔진:** 마켓플레이스 썸네일/턴테이블 프리뷰가 1차 목적이므로 **three.js**(웹, 설치 장벽 0, `<model-viewer>`로도 즉시 임베드 가능)를 프리뷰 타깃으로 확정. 실제 게임 런타임은 **Godot 4**(무료, GLB 네이티브, LightmapGI 내장, 경량 스타일라이즈드에 최적) 권장. 대안: Unity — 팀에 기존 숙련자가 있을 때만; Unreal — 이 폴리 규모·무광 스타일에 과체중.
- **포맷:** **glTF/GLB 단일 표준**(버텍스 컬러 + KHR_materials_emissive_strength + 인스턴싱 확장). VOX는 저작 중간 포맷으로만, FBX는 배제(버텍스 컬러·라이선스·툴 편차 문제).
- **폴리 예산** (베벨 박스 1개 ≈ 44tri 기준, 병합·인스턴싱 후):
  - 소형 프롭: 100–500 tri · 대형 프롭/나무: 500–2,000 tri
  - 건물 모듈: 1,000–4,000 tri · 히어로 구조물(세계수, 갤리온): 15,000–30,000 tri
  - 캐릭터(휴머노이드): 1,500–3,000 tri · 대형 크리처: 5,000–10,000 tri
  - **디오라마 씬 전체: 200k–500k tri** — 나뭇잎·군중을 GPU 인스턴싱하면 웹에서도 드로우콜 100 이하, 60fps 무난.
- **텍스처링:** **PBR 맵 없음.** 버텍스 컬러 + 씬별 16~32색 팔레트(scenes.json의 팔레트 문자열에서 생성)를 단일 256px 팔레트 아틀라스로. 머티리얼은 3개로 통일: ① 블록 공통(무광, roughness 0.65–0.8, metallic 0) ② emissive(랜턴·창문·네온·룬) ③ 받침대(유광 검정, roughness 0.05 + 프리뷰에선 평면 반사/SSR). 베벨 엣지 하이라이트는 지오메트리 베벨의 노멀이 키라이트를 받아 자연 발생 — 별도 맵 불필요.
- **리깅:** **강체 블록 스키닝(rigid per-block skinning)** — 각 블록을 정확히 1개 본에 100% 웨이트(변형 없음, 블록이 휘면 룩 파괴). 휴머노이드 본 12~16개(손가락 없음). 애니메이션은 12fps 스텝 보간으로 스톱모션 미니어처 감성을 내면 블록 룩과 시너지. 슬라임류는 본 대신 스쿼시 블렌드셰이프 2~3개.

## 4. 조명·렌더 매칭 — 권장: **베이크드 GI + 실시간 키라이트 + 고정 포스트 스택**

디오라마는 정적이므로 베이크가 정답이다.

- **GI:** Blender Cycles에서 라이트맵 베이크(또는 Godot LightmapGI) — 웜 키라이트(색온도 ~3500K, 각도 낮게)의 바운스와 랜턴의 로컬 글로우를 베이크에 포함. 실시간 GI(SDFGI/Lumen류)는 이 규모에서 불필요하고 웹에선 불가능.
- **키라이트:** 베이크와 별도로 실시간 디렉셔널 1개를 같은 각도로 유지 — 캐릭터/동적 오브젝트의 그림자와 베벨 엣지의 스펙큘러 하이라이트(레퍼런스의 블록 윤곽선)를 담당.
- **랜턴:** 전 랜턴 emissive 머티리얼(strength 3~8) → 블룸이 글로우 생성. 실제 point light는 씬당 히어로 4~6개만(분수 주변, 대문 등), 나머지는 베이크 기여로 충분.
- **림라이트:** 씬 뒤쪽 상단에서 차가운 색(청록/보라) 디렉셔널 1개, 그림자 없음 — 레퍼런스의 실루엣 분리를 재현.
- **포스트 스택(three.js: pmndrs postprocessing / Godot: WorldEnvironment):** ACES 톤매핑 → 블룸(threshold ~1.0, emissive만 반응) → **틸트시프트 DOF**(화면 상·하단 그라디언트 블러 — 물리적 DOF보다 싸고 미니어처 감이 정확) → 비네트 약하게 → 채도 +10%. 배경은 지오메트리 없이 어두운 그라디언트 환경색(레퍼런스의 검은 무대).
- **마켓플레이스 스틸컷:** 실시간 결과가 2D 레퍼런스의 95%에 도달하기 어려우므로, 썸네일 정지 이미지는 동일 씬을 **Cycles 오프라인 패스트레이싱으로 렌더**해 품질 상한을 확보하고, 웹 인터랙티브 뷰는 실시간 버전을 쓰는 이원화를 권장.

## 5. 워크플로우·툴링 — 첫 마일스톤: **pet-ranch 버티컬 슬라이스**

**선정 이유:** pet-ranch는 ① 지형+물+길(베이스 킷) ② 건물 2동+울타리(건축 킷) ③ 나무·덤불·꽃(자연 킷) ④ 슬라임·드래곤 3크기(크리처 대표) ⑤ NPC 1명(모듈러 휴머노이드) ⑥ 랜턴+골든아워(라이팅 스택 전체)를 모두 포함하면서, town-square의 군중 물량이나 fireworks-festival의 파티클 난이도가 없다. 즉 **파이프라인 전 구간을 최소 물량으로 관통**하는 씬이다.

**단계별 툴체인:**
1. `vox2blocks` 변환기 작성(Python + trimesh 또는 Rust): .vox 파싱 → greedy box decomposition(병합 상한 8) → 챔퍼 박스 지오메트리 생성 → 버텍스 컬러 → GLB 출력. 골든 테스트: 단일 8×4×2 덩어리 → 박스 1개 44tri.
2. `palette-gen`: scenes.json의 팔레트 문자열("mint and butter-yellow") → 16색 램프 → MagicaVoxel .pal 파일 자동 생성. 저작 시작 전에 팔레트를 고정해 2D 레퍼런스와 색 일치 보장.
3. MagicaVoxel로 pet-ranch 킷 저작: 헛간, 오두막, 울타리 5종, 나무 3종, 여물통, 랜턴 2종, 슬라임 3색, 베이비드래곤, 대형 드래곤, NPC 1(파트별) — 약 20에셋.
4. Blender: GLB 임포트 → 씬 어셈블(레이아웃을 JSON으로도 기록) → 머티리얼 3종 지정 → 캐릭터 리깅(idle 애니 1개) → 라이트 세팅 → Cycles 라이트맵 베이크 + 스틸컷 렌더.
5. three.js 뷰어(120줄 내외): GLB 로드 + 환경 그라디언트 + 포스트 스택 + 궤도 카메라(각도 제한: 등각 ±20°). 2D 레퍼런스와 나란히 놓고 A/B 비교 → 조명 파라미터 수렴.
6. Godot 4 임포트 스모크 테스트(같은 GLB가 게임 런타임에서도 성립하는지 확인).

**공수/비용:** 1인 기준 — 툴링(1·2·5단계) 4~6일, 에셋 저작 5~7일, 어셈블·라이팅·수렴 3~4일 = **총 2.5~3.5주**. 소프트웨어 비용 0원(MagicaVoxel/Blender/Godot/three.js 전부 무료), 비용은 사실상 인건비뿐. 이후 씬당 한계 공수는 킷 재사용률 상승으로 3~5일까지 감소 예상.

**자동화 대상:** ① `vox2blocks`(핵심, 없으면 룩 자체가 성립 안 함) ② scenes.json → 팔레트 ③ 씬 레이아웃 JSON → three.js/Godot 씬 빌더 ④ 라이트맵 베이크 헤드리스 배치(blender -b) ⑤ 턴테이블 → webp 썸네일 렌더 파이프(기존 `cwebp -q 82` 관행에 접속).

## 6. 리스크와 완화

1. **AI 이미지의 조명은 실시간으로 완전 재현 불가** — 레퍼런스는 물리적으로 불가능한 바운스·블룸을 가짐. → 완화: 스틸컷은 Cycles 오프라인 렌더로 이원화(§4), 실시간은 "95% 무드"를 목표로 A/B 수렴. 버티컬 슬라이스에서 갭을 조기 측정.
2. **소형 블록 + 베벨로 인한 폴리 폭발** — 나뭇잎 클러스터 하나가 수천 tri. → 완화: 나뭇잎·군중·자갈은 GPU 인스턴싱, 원거리 LOD는 베벨 제거(플레인 박스, -60% tri), 병합 상한 파라미터로 씬별 밀도 조절.
3. **세분화 캐릭터의 애니메이션 비용과 품질** — 블록이 관절에서 벌어지거나 겹침. → 완화: 강체 블록 스키닝 + 관절부에 의도적 갭(레퍼런스도 블록 사이 틈이 보임), 12fps 스텝 애니로 "미니어처 피규어" 컨벤션화. 애니는 idle/walk/emote 3종만으로 시작.
4. **2D 레퍼런스 간 블록 스케일 불일치** — 32장의 AI 이미지는 서로 그리드가 다름. → 완화: 버티컬 슬라이스에서 "블록 그리드 바이블"(1유닛 실측, 캐릭터 키 = N유닛, 문 높이 = M유닛)을 확정하고 이후 **3D 킷이 새 SSOT**가 됨을 수용 — 2D는 무드 레퍼런스로 강등.
5. **물·폭포·불꽃·안개 등 볼류메트릭** — 블록 문법과 이질적. → 완화: 전부 스타일라이즈드 셰이더/카드로(폭포 = 스크롤 UV 블록 기둥, 불꽃놀이 = 포인트 스프라이트 파티클, 안개 = 알파 카드). 시뮬레이션 금지.
6. **AI image-to-3D에 대한 과신** — Tripo/Meshy 산출물을 최종 에셋으로 쓰면 블록 줄눈이 뭉개져 룩이 즉시 붕괴. → 완화: §1대로 블록아웃 참고용으로만 역할 제한, 최종 지오메트리는 항상 `vox2blocks` 경로를 통과시킴.
7. **유광 받침대의 실시간 반사** — 웹에서 SSR 비용. → 완화: 받침대는 평면이므로 평면 반사(planar reflection) 1패스로 저렴하게 해결, 저사양은 큐브맵 근사로 폴백.

**결론 한 줄:** 복셀로 저작하고 박스 분해로 "비균일 블록"을 기계적으로 얻는 `vox2blocks` 파이프라인이 이 룩의 성립 조건이며, pet-ranch 버티컬 슬라이스 3주로 파이프라인 전 구간과 조명 매칭 갭을 동시에 검증하는 것이 최단 경로다.
